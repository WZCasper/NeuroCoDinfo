# NeuroCoDinfo

Новостной сайт о Call of Duty и Activision на русском языке.
Чёрно-жёлтая тема, закреплённые новости в шапке, лайки/дизлайки, комментарии с ответами, кнопки «поделиться», «копировать ссылку» и «копировать текст».

## Что уже реально работает (Этапы 1, 2, 3, 4, 5 (частично), 6)

- Главная страница с закреплённым блоком новостей и лентой с фильтрами по категориям.
- Страница отдельной статьи.
- Лайки и дизлайки — на статьях и на комментариях.
- Комментарии без авторизации, с ответами (вложенные ветки).
- Кнопки «поделиться», «скопировать ссылку», «скопировать текст статьи».
- Адаптивная вёрстка, доступность (видимый фокус, уважение `prefers-reduced-motion`).
- **Автоматический сбор новостей** с официального блога Call of Duty (`callofduty.com/blog`) по расписанию через GitHub Actions, с переводом на русский и автоматическим закреплением трёх самых новых статей.
- **Анкета игрока** (`profile.html`): имя, Activision ID, страна, дата начала игры (с автоматическим подсчётом игрового стажа), любимые режимы, платформа. Карточка профиля отображается, если анкета заполнена.
- **Telegram-бот для рассылки новостей**: сам обнаруживает, в какие группы/каналы его добавили, и рассылает туда новые статьи по расписанию. Кнопка «Подключить бота» на главной странице — готовая ссылка-приглашение.
- **Общий backend на Supabase** (опционально, см. ниже): как только он настроен, лайки, дизлайки и комментарии становятся видны всем посетителям сайта, а не только в одном браузере. Без настройки сайт продолжает работать в локальном демо-режиме — ничего не ломается.
- **Вход через Telegram** — настоящий, рабочий: виджет Telegram Login + проверка подписи на сервере (Supabase Edge Function). После входа имя и аватар берутся из Telegram и автоматически попадают в анкету. **Входа через Activision нет и не может быть** — у Activision нет публичного OAuth для сторонних сайтов (см. раздел ниже), это не баг и не недоделка, это реальное ограничение площадки.

**Важно:** по умолчанию (без какой-либо настройки) сайт работает в demo-режиме на `localStorage` — лайки и комментарии сохраняются только в вашем браузере и не видны другим посетителям. Это и есть рабочее состояние «из коробки». Если вы настроите Supabase (см. раздел ниже), всё автоматически переключится на общий для всех посетителей режим — без единой правки в HTML или в логике рендера.

Вход через Telegram настоящий и рабочий (Этап 4) — но заработает только после того, как настроены и Supabase (Этап 3), и сама Edge Function (см. соответствующий раздел ниже). До этого на месте кнопки входа отображается пояснение, что вход появится позже.

## Как запустить локально

Открывать `index.html` просто двойным щелчком **не сработает** — браузер заблокирует загрузку `data/posts.json` из соображений безопасности (CORS для `file://`). Нужен любой простой локальный сервер:

```bash
# вариант 1, если установлен Node.js
npx serve .

# вариант 2, если установлен Python
python3 -m http.server 8080
```

Затем открыть `http://localhost:8080` (или адрес, который покажет команда).

## Как выложить на GitHub Pages

1. Создайте новый репозиторий на GitHub (например, `neurocodinfo`).
2. Загрузите туда все файлы из этой папки (через `git push` или просто перетащив файлы в веб-интерфейсе GitHub).
3. В репозитории откройте **Settings → Pages**.
4. В разделе **Source** выберите branch `main` и папку `/ (root)`, нажмите **Save**.
5. Через 1–2 минуты сайт будет доступен по адресу вида `https://ваш-логин.github.io/neurocodinfo/`.

## Как настроить автосбор новостей (Этап 2)

Пайплайн уже работает по умолчанию даже без дополнительной настройки — он будет собирать заголовки и короткие описания статей **на английском**, без перевода. Чтобы включить перевод на русский:

1. Зарегистрируйтесь на [deepl.com/pro-api](https://www.deepl.com/pro-api) и получите бесплатный API-ключ (Free-тариф — до 500 000 символов в месяц, для новостного сайта этого более чем достаточно).
2. В репозитории на GitHub откройте **Settings → Secrets and variables → Actions → New repository secret**.
3. Создайте секрет с именем `DEEPL_API_KEY` и вставьте туда ваш ключ.
4. В **Settings → Actions → General → Workflow permissions** выберите **Read and write permissions** — иначе бот не сможет закоммитить новые статьи обратно в репозиторий.

Дальше пайплайн запускается сам каждые 3 часа. Проверить и запустить вручную: вкладка **Actions** репозитория → workflow «Сбор и перевод новостей» → **Run workflow**.

**Откуда берутся новости и почему именно так:**
- Источник — `callofduty.com/blog`. У него нет официального RSS или публичного API (это проверено), поэтому скрипт `scripts/fetch-news.mjs` аккуратно разбирает HTML страницы блога (она отдаётся сервером уже готовой, без обязательного JavaScript).
- Публикуются **короткие переведённые описания** статей (1–2 предложения, из метатега страницы) и ссылка на первоисточник — **не полный текст** статьи. Это сделано специально, чтобы не нарушать авторские права.
- Раздел «Эспорт» (`callofdutyleague.com`) в автопайплайн **не входит**: этот сайт подгружает новости через JavaScript, и для него нужен headless-браузер (Playwright/Puppeteer) — это можно добавить отдельно позже.
- Если Activision изменит структуру страницы блога, разбор может временно перестать находить статьи. Это нормальная особенность любого решения, основанного на разборе HTML, а не на официальном API. Чинить — смотреть лог запуска во вкладке Actions и поправить селекторы в `getCandidateLinks()` внутри `scripts/fetch-news.mjs`.

## Как подключить общий backend на Supabase (Этап 3)

1. Зарегистрируйтесь на [supabase.com](https://supabase.com) и создайте новый проект (бесплатный тариф подходит).
2. В дашборде проекта откройте **SQL Editor → New query**, вставьте туда содержимое файла `supabase/schema.sql` целиком и нажмите **Run**. Это создаст таблицы для комментариев и реакций, включит защиту на уровне строк (RLS) и базовую защиту от спама.
3. Откройте **Settings → API Keys** (или диалог **Connect** на главной странице проекта). Скопируйте:
   - **Project URL** — выглядит как `https://abcdefgh.supabase.co`;
   - публичный ключ — в зависимости от того, когда создан проект, он может называться **anon key** (длинная строка вида JWT) или **Publishable key** (вид `sb_publishable_...`) — подходит любой из них, оба специально предназначены для браузера.
4. Вставьте оба значения в `js/config.js`, в поля `supabaseUrl` и `supabaseAnonKey`.
5. Готово — после этого комментарии и лайки/дизлайки автоматически становятся общими для всех посетителей сайта. Никаких других изменений в коде не требуется: `js/store.js` сам определяет, что Supabase настроен, и переключается на него.

**Честно о пределах этого решения:** пока нет настоящей авторизации (Этап 4), у каждого посетителя есть случайный анонимный идентификатор браузера (`device_id`/`voter_id`), а не подтверждённый аккаунт. Это значит, что технически продвинутый человек может подделать чужой идентификатор и, например, снять чужой лайк. Простая защита от спама на комментарии (не больше одного раз в 10 секунд с одного устройства) есть, но это не замена настоящей модерации. Когда появится Этап 4, `voter_id` стоит заменить на `auth.uid()`, а политики доступа в `supabase/schema.sql` — сделать строже (это прямо описано комментариями внутри самого файла).

## Почему нет входа через Activision (Этап 4)

Изначально планировалась авторизация через несколько соцсетей, но по вашему решению список сократился до Activision и Telegram. С Telegram всё в порядке (см. ниже), а вот с Activision — нет, и вот почему.

Я специально проверил это перед тем, как писать код: у Activision **нет публичного OAuth-провайдера**, на который мог бы зарегистрироваться сторонний сайт — в отличие от VK, Google или Telegram, где это можно сделать самостоятельно за пару минут. У Activision есть API, но доступ к нему — только через программу **«approved partners and licensees»**, то есть через официальное партнёрство с компанией, а не через форму регистрации приложения. URL вида `s.activision.com/cod/oauth/consent`, который можно найти в сети, — это **внутренний** механизм Activision для привязки PlayStation/Xbox/Steam/Battle.net аккаунтов к их собственному аккаунту, а не публичный вход для сторонних сайтов.

Что я не стал делать: имитировать страницу входа Activision и просить посетителей вводить туда логин и пароль от настоящего аккаунта. Это было бы фактически фишингом — даже без злого умысла, это небезопасный паттерн, который может стоить игрокам аккаунта, и я не стану его реализовывать ни в каком виде, даже как «временную заглушку».

Что есть вместо этого: поле **Activision ID** в анкете (`profile.html`) — игрок вписывает свой тег сам, без проверки. Это честное и безопасное решение в рамках того, что вообще доступно независимому сайту.

## Как настроить вход через Telegram (Этап 4)

1. Откройте [@BotFather](https://t.me/BotFather) и отправьте `/setdomain` — выберите вашего бота (того же, что создали для Этапа 6) и укажите домен сайта (например, `ваш-логин.github.io`, без `https://` и без конечного слеша). Без этого шага виджет Telegram будет показывать ошибку про недопустимый домен.
2. Установите [Supabase CLI](https://supabase.com/docs/guides/cli) и выполните `supabase login`, затем `supabase link` (привяжет CLI к вашему проекту).
3. Из папки проекта выполните:
   ```
   supabase functions deploy telegram-login
   ```
   Файл `supabase/config.toml` уже настроен так, чтобы эта функция не требовала стандартной проверки Supabase JWT (это нормально для неё — она проверяет подлинность по-своему, через подпись Telegram). Если после деплоя всё равно видите ошибку `401 Missing authorization header` — у CLI иногда не применяются настройки `config.toml` к уже существующей функции (известная особенность); в этом случае откройте в дашборде **Edge Functions → telegram-login → Settings** и выключите проверку JWT там вручную.
4. Добавьте секрет для самой функции — **отдельно от секрета GitHub Actions**, это разные хранилища, хотя значение токена одно и то же:
   ```
   supabase secrets set TELEGRAM_BOT_TOKEN=ваш_токен_от_BotFather
   ```
5. Готово. На сайте кнопка входа через Telegram появится автоматически, как только настроен Supabase (Этап 3) — до этого момента на её месте отображается пояснение, что вход появится позже.

**Что происходит технически:** виджет Telegram передаёт подписанные данные напрямую в браузер, мы отправляем их в Edge Function, она проверяет подпись секретным токеном бота (в браузер токен никогда не попадает) и возвращает подтверждённые имя/аватар/id. Эти данные сохраняются как устойчивая личность посетителя — она используется и для анкеты, и как идентификатор при лайках/комментариях вместо случайного идентификатора браузера.

**Честно о пределах:** это делает личность посетителя подтверждённой и устойчивой (не теряется при очистке localStorage, одинакова на разных устройствах с одним Telegram-аккаунтом), но **не превращает её в полноценную сессию Supabase Auth** — политики доступа в `supabase/schema.sql` остаются такими же открытыми, как и для анонимных посетителей (см. пояснение в самом файле). Полная привязка к `auth.uid()` технически возможна, но требует либо подписывать собственный JWT секретом проекта Supabase (а с октября 2025 новые проекты по умолчанию используют асимметричные ключи, и для них этот способ работать не будет — я не стал на нём строить решение, чтобы не выдавать недостоверно работающий код за надёжный), либо использовать новый протокол Telegram OIDC (`core.telegram.org/bots/telegram-login`) через нативную поддержку «custom OIDC providers» в Supabase Auth — это многообещающий, но совсем новый путь, который я не реализовал, потому что не могу проверить его в реальном проекте из этой среды. Если решите усиливать защиту дальше — стоит исследовать именно этот вариант.

## Как настроить Telegram-бота для рассылки (Этап 6)

1. В Telegram напишите [@BotFather](https://t.me/BotFather), отправьте `/newbot` и следуйте инструкциям — получите токен вида `123456:ABC-DEF...` и username бота (например, `NeuroCoDinfoBot`).
2. В репозитории: **Settings → Secrets and variables → Actions → New repository secret** → имя `TELEGRAM_BOT_TOKEN`, значение — токен от BotFather.
3. (Необязательно) Если у сайта свой домен, а не стандартный `*.github.io` — добавьте переменную (не секрет) `SITE_URL` там же, на вкладке **Variables**, со значением вида `https://ваш-домен.ru/`. Если не указать — бот сам вычислит адрес по умолчанию из имени репозитория на `github.io`.
4. В `js/config.js` впишите `telegramBotUsername` — без `@`. Это публичное имя бота, не секрет, нужно для кнопки «Подключить бота» на сайте.
5. Готово. Любой желающий может нажать кнопку на сайте, добавить бота в свою группу или канал и **сделать его администратором с правом публикации сообщений** — без этого права бот не сможет постить.

Бот проверяет, куда его добавили/откуда убрали, и рассылает новые статьи каждые 15 минут (см. `.github/workflows/telegram-broadcast.yml`). Список чатов и история рассылок хранятся в `data/telegram-state.json` — туда не нужно лезть руками, бот сам его обновляет и коммитит изменения.

Если в логе запуска (вкладка Actions) видите ошибку про «webhook is active» — значит для этого бота где-то раньше включили вебхук. Откройте один раз в браузере `https://api.telegram.org/bot<ВАШ_ТОКЕН>/deleteWebhook` и запустите workflow заново.

## Лицензия / правовая оговорка

Это неофициальный фан-проект. Сайт не связан с Activision Blizzard. Автопайплайн новостей (Этап 2) публикует короткие переведённые выдержки со ссылкой на первоисточник, а не полные тексты статей — это и безопаснее по авторскому праву, и стандартная практика новостных агрегаторов. Если будете добавлять другие источники — придерживайтесь того же принципа.
